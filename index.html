Smashing Git Cheat Sheet

An Explanation

This Cheatsheet should help you getting started with Github and on how to use it the right way.

Where one may argue, that there's never only one right way, for Git this is true as well but... yeah, you knew that but would come, right?

Of course there is more than one way to work with Git. You can use the Command Line, there are a few GUI-Applications you can download (some are free, some not. I'd recommend trying a free one first) and if nothing else you can go right to Github and work it from there.

With that being said, let's dive in together, shall we?

Why Git

Git is cool when you are developing. Whether it is a website or an application or some piece of software or even this cheatsheet your are reading, now. You could write books with versioning or keep your favorite recipies at some repository (from now on: repo).

Why is it cool? Because you will never lose any data while developing your whatever and if something goes wrong you can simply take a step back to your last working version and go on from there or start over completely at any time.

If you're still not convinced: with Git you can add some change to your stuff locally, commit it with a single command to your repo at Github and automatically push those changes to your server. Smells like large amounts of WIN, doesn't it.

The Right Way

Using Git the right way, would be using Git with Terminal or if you want to call it like that "The Command Line Tool".

Now, what can we do with Git and Terminal? We can clone an existing remote repo at Git to your computer, (we could also create a completely new one on your computer,) we can look for changes you applied, we can stage those changes, we can then commit them with a short message and finally push them to the repo at Git.

Those are the basic tools you are going to need, when working with our repos at Git and on your computer. While you might feel intimidated when using the Terminal for the first time, as you've been told that great evil can occur by doing so, I can assure you that those people did not lie to you.

"With great power comes great responsibility!" This is a quote by Uncle Ben from the movie Spiderman and boy is he right.

You could, when prompting the wrong command (or the right one, that depends on the way you look at it), easily erase all files from your hard-drive or even better from any exisiting web-server you have access to.

Yet, I won't tell you the command and you will not look for it. Deal? Deal! And promise me to not take commands by "some dude on stackoverflow" and copypaste them to your Terminal. Okay? Thank you.

Well, then I suppose we're safe from doing such terrible things and move on to the very nice stuff Terminal comes with.

Your First Commands

To give you an easy start, we are going to create a directory, move into it and then clone an existing repository from the Smashing Accont at Git. Shall we? We shall! I can already feel your excitement.

Let's create a folder on your computer by typing:

mkdir my-git-directory

You may then hit enter. mkdir means nothing else than "Please, dear Terminal, create a directory". Important! Name this folder anything you want but please: NO SPACES! The internet is not playing nicely when it comes to spaces.

We have a folder! We want to go in there! Terminal to the rescue:

cd my-git-directory

And again hit enter, please. At that point you might wonder why noone is telling you:"Good job!" or "Directory created." or "We successfully moved into that folder.". Terminal will only tell you, if something goes wrong. How sweet. Try:

cd whatever

Now, that you are inside my-git-directory, which should be empty at this point in time you will be prompted with an error.

Error? Error! Good thing. We are up to fill this folder with precious stuff in a breeze.

cd

means nothing else but change directory.

You are now inside your wonderful git folder and you want to clone a repo from Git. That is also very easy. After making sure you have the rights to do this (if you're reading this, you should already have those rights), you type in Terminal the following:

git clone git@github.com:smashingmagazine/smashing.cat.git

and hit enter. Then you wait for the download to be ready while stroking your geek-beard. If you haven't got a geek-beard yet, don't fear, it will grow automatically – in minutes.

Important Note: While typing please make sure you watch out for case-sensitivity. Otherwise things will go awry and we don't want that. Again! Always watch out for the exact name of a repo or you will end up with either nothing or something you did not want to have on your computer.

Down The Rabbit Hole

You have successfully cloned a repo to your computer. Hooray! You are already on a very good path to work with Git. Dive into the project directory:

cd smashing.cat

Again, watch out for case-sensitivity or you will end up nowhere.

Go on and open the index.html of the smashing.cat repo in your favorite editor and type away. Add something, delete something, change something and safe those changes! For example you could change the title to something to your own likes.

Back to the Terminal Batman! We made a change. May be we made a few or even many changes. May be we have a completely new version compared to the one you cloned from Git. Whatever it is, you now want to type:

git status

The Terminal will now tell you what has changed in your local repo.

Like what you see? Good. Now, we will add those changes to the staging area by typing:

git add .

and tap return. The dot at the end of this command means nothing else than ADD ALL THE THINGS that changed to the staging area. Check again with:

git status

It is not a mandatory step to do this but helpful nonetheless as you can check what has been staged.

Now, we want to commit out changes to our local repo and give a message what we have changed so we can remember the next time we come back to it.

git commit -m "Your message goes here"

This is what you want to type before pushing down the enter-button once more. The command tells this: "Commit all my changes and give a message with it. The message is between the quotation marks."

That's it. You made a complete circle of version control. Check with:

git status

All things have been committed? If that is the case we move on to our final section of this little course.

The Pusher

You've come this far! Wow. Give yourself a nice pat on the back, now.

The final step of working with Git is to push your changes to the repo at Github. Back to the terminal I say and you should type the following command:

git push -u origin master

Hit enter and take a look at your repo at Github. Sure enough there's +1 Commit, now, in the history of the remote repository. You did it.

Now, what if someone else made a few changes to that repo after your committing and pushing and all? That's also as easy as falling off a log. cd to your repo on your computer, hit the following command:

git fetch git://github.com/smashingmagazine/smashing.cat

Followed by the obligatory enter button.

Git will now check the differences between your repo and the one at Git. All new things, will then be downloaded to your computer. Type:

git merge git://github.com/smashingmagazine/smashing.cat/master

You just merged the changes with your repo.

Marvelous stuff it is. You learned how to clone a repo, how to check for changes, how to add changes to the staging area, commit them and finally push them to the remote repo at Git. Pretty impressing! And on top of that you now can keep your repo always up to date with recent changes committed by others, if you're on a team-project.

tl;dr

Too long, didn't read. Alright, here is the very short commands only version of this course/cheatsheet, whatever you want to name it. Have fun, always double check and try to be consistent! Nuff said, here are the most needed commands:

git clone [alias]
git status
git add .
git commit -m "Your message here"
git push -u origin master
git fetch [alias]
git merge [alias]/[branch]

For a complete overview of what is what and who is who check the Git Reference!
